Primary Key:
=============

Primary key is a constraint
Primary constraint is the combination of two constrains
1)UNIQUE
2)NOT NULL

When we use primary key constraint for a column then that column value shouldn't be null and it should be unique.

it is not recommended to set values for primary key column manually

we will use generators to generate the value for primary key column

To use Geneator we will use @GeneratedValue annotation.

In mysql database we will use auto_increment to generate value for primary key. To specify auto_increment we will use IDENTITY generator.

		@Id
	@Column(name="Book_ID")
	@GeneratedValue(strategy = GenerationType.IDENTITY)
	private Integer bookid;
	
In ORACLE database we will use sequence concept to generate value for primarykey column. For every primary key one sequence will be created in database.

if we want to generate primary key column value like below then we shoud go for custom generator(Our own generator we have to develop)

INF01

Restful Services:

========================================

Rest stands for Representational state transfer
Restful services are used to develop distributed application
Distributed applications are used for B2B Communications
B2B means Business to Business communication
If one application is communicating with another application then those are called Distributed application
Distributed application will re-use service of one application in another application
Restful services are providing "Interoperabiility"

Interoperabiility means irresepective of the platform and language application can communicate.

Two Actors will be involved in Distributed application development
	provider
	Consumer
	
Provider means which will provide business services is called Provider Application
The application which is accessing business service is called as consumer application

Note: One provider application can have multiple consumer application

Provider and Consumer will communicate using HTTP as a mediator.
Provider and Consumer will exchange the data in Text/XML/JSON Format


To start our journey with restful services development we should be good in below areas

Http protocols(Methods, status codes, Req Structure, Res Structure).
xml and JAX-B API(java architecture for xml binding)
JSON and jackson API


What is HTTP?

HTTP Stands for Hyper Text transer protocol.
It is used as a mediator between client and server.(Consumer & Provider)
Consumer application will send HTTP Request to provider Application.
Provider application will process the request and it will send HTTP Response to the consumer.

Http Methods:
=========================

Http Methods are used to send request from consumer application to provider application.

Http method will represent what type of operation client/Consumer wants to perform with provider

###
GET
POST
PUT
DELETE

Get request is used to retrieve data from provider application.
Get request will not have body
To Send any data from consumer to providing using GET request then we will use Path params and Query params.

Note: Path Params and Query params data will displayed in the URL.

Note: It is not recommended to send sensitive/secret  data using PATH Params and Query Params.

Get request is Idempotent (Means if you send same GET request for multiple times also nothing will change at server).


===========================================

Post Request is used to create a new record at server
When consumer wants to sends huge data or sensitive data then consumer will use POST request.
post request contains request body
Post request is non-idempotent


Note: In Post Request we can send the data in URL and Request Body.

Note: Request Body is the recommended approach to send sensitive data to server.

=========================================================================

PUT
PUT request is used to update a record at a server
When consumer wants to update a record at then consumer will send PUT request to provider 
PUT request contains request body
PUT request is non-idempotent

Note: In PUT request we can send in URL and in Request Body.
Note: Request body is the recommended approach to send sensitive data to server.


===============================================================

DELETE

DELETE request is used to delete a record at server
DELETE request contains request body
DELETE request is non-idempotent

JAX-B : Java architecture for XML Binding.

Note: Earlier people use to create Binding Classes using XSD, XSD represents structure of xml.

Control marshalling and unmarshalling using fields of entity class.

@XmlAccessorOrder :==Follow order of variables in the class to marshall and unmarshall.

@XmlElement(name ="PhoneNo") : It is optional
it is the element name that it will come by default with variable name if we declared it as @XmlElement then we need to specify the name as element.It is used to change the name of the element.


Note: By Default every variable is considered as element and variable name will be considered as element name.

@XmlTransient ==== To skip a variable in marshalling.

->Actuators are giving production ready feature for our boot application.

Actuator Endpoint:
==================
/health: To get application health status
/info : To get application information.
/beans: To get what beans loaded by our application.
/mappings: To get what URL patterns available in our application.
/configProps: To get configuration properties loaded by our application.
/heapdump: To download heap data
/threaddump: To get threads information
/shutdown: To stop our application (This is special, it is binded to POST request)

To work with actuators we have to use "spring-boot-starter-actuator" dependency.

Note: /health is a default endpoint which we can access directly.

We can expose other actuator endpoints using below property.

	management						
	 endpoints
	  web
	   exposure
	    include or :'*'
		 -beans
		 -mappings
		 
		 We can also use exclude also exclude:beans

Microservices:
==============
Application will be developed in 2 ways
1)Monolithic Architecture
2)Microservices Architecture

if we develop all the functionallities in one single application then it is called as monolithic application.

Advantages:
===========

Development is easy
Deployment is easy
performance
Easy Testing
Easy Debugging
KT is easy

Disadvantanges:
===============

Single point of failure
Whole application Redeployment
Scalability(Increasing and Decreasing resources based on demand)
Reliability(Strong)
Availability(Zero Downtime)


===>If we develop the functionallities in multiple service/apis then it is called as Microservices Architecture based application.

Every microservices will have its own goal.

Advantages of microservices:
============================

Loosely coupled
Fast Development
Quick Releases
Flexibility
Scalability
No single point of failure
Technology independence

Challenges:
==============
Bounded Context(identifying no of services to develop)
Lot of configurations(Database for every service we need to configure in every service)
visibility(not get a chance to work with all the services)
Testing is difficult
Debugging

Microservices Architecture:

===========================

Microservices is a design pattern to develop applications.

There is no fixed architecture for microservices based application

people are customizing Microservices architecture according to their requirement.


=============================================

Let us see generalized architecture of microservices.*************

1)Service Registry
2)Admin Server
3)Zipkin Server
4)Services(Rest API'S)
5)FeignClient
6)API Gateway


Service Registry:Eureka server
================	
In microservices, many services run on different ports/IPs.

Example:

User Service → 8081
Order Service → 8082
Payment Service → 8083

If port changes dynamically, how will one service find another?

Service Registry solves this.

All services register themselves.

USER-SERVICE → registered
ORDER-SERVICE → registered
PAYMENT-SERVICE → registered

Then other services ask registry.


Admin Server:It
============
 is used to monitor our application at one place (Collection of all services) and provide the user interface.
 
 
	Used to monitor all microservices in one dashboard.

Shows:

UP / DOWN status
Memory usage
CPU
Logs
Health checks
Environment properties
Real-Time Example

If Payment Service crashes, Admin Server shows RED / DOWN.


Zipkin Server:It is used to trace our application execution details distributed log tracing we will understand by using zipkin server. Which request received how much time taken by server for process the request each execution details. 


What is use?

Tracks one request across many services.

Example:

Client places order:

Gateway → Order Service → Payment Service → Inventory Service

Zipkin shows:

Which service called
Time taken
Where delay happened
Full request path
Real-Time Benefit

If response slow, identify bottleneck.


Services: Services are nothing but rest api's are developed in project. it will act as a client for Service Registry.The actual business logic present in the services.

 
 
FeignClient:it is used for interservice communication.One api is communicating with another api within the project.Our services will talk to other services is called as External service communication.


API Gateway:Will act as a mediator between users to the backend. Api gateway will act as a front gate for all the api's.
	(Filters+Routing).
	
	
	
	Service Registry: Service Registry is used to register all our backend services/api
	Service Registry will maintain services names , urls and status of each services.
	We can use EurekaServer as a service Registry.
	
	Note: Eurekaserver provided by spirng cloud Netflix Library.
	
	-AdminServer is used to monitor and manager all our backend services at one place.
	-AdminServer will provide user interface to monitor and manager our service
	-Using AdminServer user interface we can access Acutator Endpoints of our services at one place.
	
	Note : Adminserver and Adminclient provided by "code centric" company (we can integrate with boot)
	
	
	-ZipkinServer is used for Distributed Log Tracing
	-ZipkinServer will provide user interface to monitor application execution details.
	-How many services involved and which service took more time to process the request can be monitored using zipkin.
	
	Note: Zipking is the third party opensource server(can be integrated with springboot)
	
	-Backend Services are nothing but rest api's (Which are also called as microservices)
	-Backend Rest api's contains actual business logic of our application
	-one project will have multiple rest api in the backend.
	-Each Backend api will act as client for Service\
>,mn 	Registry+ Adin Server + Zipkin Server
	
	
	
	
	-with in the same application if one backend api communicating with another backend api then it is called as interservice communication.
	
	Feign Client will be used to perform interservice Communication.
	
	Note: Based on requirement our backend api's can communicate wit 3rd party apis using Rest Template or Webclient.
	
	
	
	
	-API Gateway will act as entry point for backend apis
	-it acts as mediator between end users and backend apis.
	-API Gateway contains Filter and Routing
	-Filters we can use to authenticate incoming requests.
	-Routing will decide which request should go to which backend api
	
	Note: Earlier we used to work with Zuul Proxy for API Gateway but now it got removed from latest version of boot
	
	Spring cloud Gateway we can use as API Gateway for our application.

Load Balancing:

================


Load Balancing is the process of distributing loads to multiple servers.




If we run our application on single server then all requests will be sent to single server
->Burder will be increased on the server
->When burden got increases request processing gets delayed

->Sometimes our server might crash due to heavy loaded


***********To overcome the problem*************************

-> Load Balancing is the process of distributing load to multiple servers

Scaling:
============

Scaling can be done in 2 ways

1)Vertical scaling: Increasing system resources(Ex: RAM,CPU,PROCESSOR ETC)

2)Horizontal Scaling: Increasing the servers to run our application.


scale up : Increasing number of servers based on demand.

scale down: Decreasing the no of servers when traffic is less.


Note : In cloud platforms like AWS , Azure, and GCP we have Auto-scaling concept.


-->Auto Sclaing means no of servers will be increased or decreased based on no of requests coming to application.



Circuit Breaker:

================================

Circuit Breaker is a design pattern.

It is used to implement fault tolerance systems.

Fault Tolerance systems are also called as resillence systems.



When m1 methods is failing contionusly ex: for 5 requests) then i want to execute only m2() method directly for next 30 mins. We can achieve this requirement by using "Circuit Breaker".

-->Cache is a temporary storage. 

When our application want to access same data frequently then we will use cache memory.

Cache will improve performance of our application.

	To reduce no of round trips between application and database we will use 'cache'.
